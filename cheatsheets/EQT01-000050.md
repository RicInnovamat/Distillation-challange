## CHEAT SHEET: EQUATIONAL THEORIES IMPLICATION (MAGMAS)
### Version 3F - Balanced Hybrid (Minimal Hard-FALSE Guard)

### 1. Task Definition (Always Keep This Fixed)
- You are given:
  - Equation 1 (assumption law): `L1 = R1`
  - Equation 2 (target law): `L2 = R2`
- Goal:
  - Decide whether `Eq1 => Eq2` over **all magmas** (all binary operations on all non-empty sets).
- Output policy:
  - If TRUE: provide a proof sketch.
  - If FALSE: provide a finite counterexample magma and verify both equations.

---
### 2. Canonical Parsing & Normalization
- Parse each term as a binary tree over `*`.
- Remove irrelevant whitespace and redundant outer parentheses.
- Canonicalize variable names by first appearance (`_v0, _v1, ...`) before structural comparison.
- Compare equations up to:
  - side swap (`A=B` same as `B=A`),
  - variable renaming,
  - alpha-equivalent tree shape.

Why: many implication wins/fails can be detected by syntax-level invariants before heavy search.

---
### 3. Fast Decision Filters (Cheap, High Impact)
### A. Immediate TRUE patterns
- Target reflexivity: `L2` and `R2` are syntactically identical (tautological law).
- Eq2 is Eq1 after variable renaming / side swap.
- Eq1 gives an immediate TRUE only when you can point to a direct reason that already forces Eq2:
  - Eq2 is literally Eq1 up to renaming or side swap, or
  - Eq2 follows by an explicit short rewrite or property chain from Eq1.
- Do not answer TRUE only because Eq1 seems powerful, restrictive, or likely to collapse many terms.

### B. Immediate FALSE patterns
- Find one finite magma where Eq1 holds and Eq2 fails.
- If a model checker/candidate search returns a verified counterexample, stop and answer FALSE.

Rule: a single valid counterexample is decisive; no probability adjustment can override it.

---
### 4. Proof-Oriented Strategy (for TRUE)
Use this order:

1. **Derivation by substitution/rewrite**
- Build bidirectional rewrite rules from Eq1: `L1 <-> R1`.
- Attempt to rewrite `L2` to `R2` (or both to a common normal form).
- If reachable, produce explicit derivation steps.

2. **Property extraction from Eq1**
- Check whether Eq1 enforces known strong families:
  - projection-like behavior,
  - absorption-like collapse,
  - commutativity/associativity/idempotence-type constraints.
- If Eq2 is a weaker consequence of extracted properties, prove via property chain.

3. **Universalization/trivialization argument**
- If Eq1 forces all terms to collapse to one value/class, then all equations in that class follow.
- State the forcing mechanism clearly (do not just claim “obvious”).

Proof style:
- Keep concise but explicit.
- Name substitutions and the instantiated variables used.

---
### 5. Counterexample-Oriented Strategy (for FALSE)
Try small magmas first (size 2 -> 3 -> 4), then structured families.
If Eq1 looks strong, but Eq2 still has one or more of the following features, prefer counterexample search before proof:
- Eq2 still contains nested products of depth at least 2.
- Eq2 still repeats variables many times.
- Eq2 still uses several interacting variables rather than collapsing to a simple identity or projection law.
- Eq2 is not an obvious renaming, weakening, or short rewrite consequence of Eq1.

### A. Candidate families to enumerate
- All 2x2 tables (exhaustive).
- Left-zero / right-zero / constant operations.
- Projections (`x*y=x`, `x*y=y`) and near-projections.
- Boolean-like tables (`xor`, `and`, `or`) on small domains.
- Small modular-linear forms (e.g., `x*y = a x + b y + c (mod n)`).
- Random and hill-climbed tables (when deterministic families fail).

### B. Validation protocol (mandatory)
For a found table, verify:
1. Eq1 is true for all assignments of its variables.
2. Eq2 is false for at least one assignment (show witness assignment).

Return format for counterexample:
- Set/domain.
- Full operation table.
- Explicit Eq1 check summary.
- Explicit Eq2 violating assignment and evaluation.

---
### 6. Heuristic Signals (Use Only as Guidance)
If proof/counterexample not yet found, prioritize search using these signals:

- **Subterm coverage**: if Eq2 subterms are mostly contained in Eq1 subterms, implication is more plausible.
- **Variable overlap ratio**: larger overlap suggests stronger relation.
- **Complexity gradient**: Eq1 structurally stronger/simpler constraints may imply a more complex Eq2 less often unless forced.
- **Law-type strength ordering** (rough): projection/degeneracy > identification/permutation > generic.
- **Rewrite reachability overlap**: if Eq2 sides fall into same Eq1-rewrite basin, likely TRUE.

Important:
- Heuristics rank candidates; they do not replace proof/counterexample.

---
### 7. Minimal Solve Pipeline (Recommended)
1. Parse + canonicalize.
2. Run fast TRUE filters.
3. Try directed rewrite proof (bounded depth).
4. Run finite model search for counterexample.
5. If undecided, use heuristic ranking to choose next rewrites/models.
6. Return strict final verdict with proof or counterexample.

---
### 8. Common Failure Modes to Avoid
- Declaring TRUE without an actual derivation chain.
- Declaring FALSE without a validated table + violating assignment.
- Mixing heuristic confidence with formal validity.
- Forgetting side swap/renaming equivalence.
- Common false-positive pattern: Eq1 looks restrictive, and the model jumps to TRUE even though Eq2 still contains extra structure, repeated variables, or deeper nesting.
- Do not do this unless you can show a concrete derivation or a verified universal forcing argument.
- Returning malformed output sections.

---
### 9. Output Template (Keep Deterministic)
Use this exact structure in final model response:

- `VERDICT: TRUE|FALSE`
- `REASONING: ...`
- `PROOF: ...`
- `COUNTEREXAMPLE: ...`

Output discipline:
- Do not add extra headers such as `MODEL_NAME:` or `OUTPUT_RESULT:`.
- Always include all four headers exactly once and in the order shown above.
- If TRUE:
  - `PROOF:` must be non-empty.
  - `COUNTEREXAMPLE:` must be empty, but still keep the header.
- If FALSE:
  - `COUNTEREXAMPLE:` must be non-empty.
  - `PROOF:` must be empty, but still keep the header.
- Before finalizing, check that no required header is missing.

---
### 10. Distilled Practice Rule
When uncertain:
- Prefer **searching for a counterexample** first.
- If no counterexample appears in strong small-model families, attempt a structured proof.
- Never return a final verdict without one of:
  - a derivation/proof witness (TRUE), or
  - a model witness (FALSE).

Output format (use exact headers without any additional text or formatting):
VERDICT: must be exactly TRUE or FALSE (in the same line).
REASONING: must be non-empty.
PROOF: required if VERDICT is TRUE, empty otherwise.
